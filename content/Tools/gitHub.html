分布式版本控制：当前流行的开源版本控制工具

  • 本地操作
    • git clone <项目地址> 克隆GitHub上的项目
    • git add <文件路径> 内容“复制”到暂存区
    • git commit 暂存区的内容“复制”到本地仓库
    • git commit -a 工作目录和暂存区的内容“复制”到本地仓库
    • git reset HEAD <文件路径> 移除暂存区的内容
    • git checkout --<文件路径> 本地仓库的内容“复制”到工作目录
    • git status 检查文件状态（在哪个区域）
    • git commit --amend 修改上次的提交（信息/内容）
    • git rm --cached <文件路径> 内容“离开”了GitHub
    • .gitignore（用正则表达式忽略匹配的 文件[夹]）
      • * 匹配任意多个字符
      • [] 匹配[]中枚举的字符
      • ? 匹配一个字符
      • */ 匹配目录
      • ! 取反
  • 分支
    • HEAD 是当前分支的别名
    • git branch [branch-name] 创建分支
    • git checkout [branch-name] 切换到分支
    • git merge [branch-name] 合并分支内容到HEAD(有时要手动解决冲突)
    • git branch --merged 列出已合并的分支(可以删了)
    • git branch -d [branch-name] 删除分支
    • 分支工作流程
      1. 创建项目，在默认的master分支上工作
      2. 遇到bug或新需求，创建并切换到新分支上工作
      3. bug或新需求完成后，切换回master，然后合并新分支
      4. 删除分支，继续在master上工作
  • 管理远程仓库（协作）
    • git remote add [远程名] [url] 跟踪远程仓库并起名(自动跟踪master)
    • git remote show [远程名] 查看指定仓库的信息
    • git remote rm [远程名] 不再跟踪指定仓库
    • git remote -v 列出所有远程仓库
    • git fetch [远程名] 将指定仓库的数据抓到本地
    • git push [远程名] [本地分支]:[远程分支] 将分支的内容推送给指定仓库
    • git checkout --track [远程名]/[分支名] 跟踪远程分支
    • git push [远程名] :[远程分支] 不再跟踪远程分支

注：
1.GitHub使用3个区域：工作目录(编辑内容)、暂存区(打算提交)、本地仓库(已经提交)管理提交过程
2.使用GitHub的过程，就是文件内容在3个区域的“行走”过程
3.远程仓库是托管在网络上的项目仓库，需要先在本地跟踪远程仓库，然后与之交换数据
4.分支可以分离项目的关注点，在master上维持稳定版本，在其它branch上专攻某些特性
5.参考书目《Pro Git（中文版）》
